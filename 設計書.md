## 1. 商品ステータス遷移(全設計の背骨)

`items.status` (integer) で取引の状態を一元管理する。

```
0 出品中 ──購入──▶ 1 投函待ち ──出品者が投函──▶ 2 受取待ち
                                                    │
                                          購入者が受け取り開始
                                          (録画開始+番号開示)
                                                    ▼
        4 取引完了 ◀──購入者が受取確認── 3 受取開始済み
        (出品者に入金)                        │
                                            通報
                                                    ▼
        6 返金済み ◀──運営が返金裁定── 5 審査中 ──運営が出品者支持──▶ 4
        (購入者に返金)
```

| 値 | 状態 | 意味 |
|---|---|---|
| 0 | 出品中 | 購入可能 |
| 1 | 投函待ち | 購入済み。代金はエスクロー(運営預かり)。出品者の投函を待つ |
| 2 | 受取待ち | 投函済み。購入者の「受け取り開始」を待つ |
| 3 | 受取開始済み | 録画開始・番号開示済み。購入者の確認(完了 or 通報)を待つ |
| 4 | 取引完了 | 出品者のポイント残高に代金が入金された終着点 |
| 5 | 審査中 | 通報あり。運営が動画を審査する |
| 6 | 返金済み | 運営裁定により購入者へ返金された終着点 |

| メソッド | パス | 機能 | 差分 |

### 2.4 マイページ・ポイント・通知

| メソッド | パス | 機能 | 差分 |
|---|---|---|---|
| GET | `/mypage` | 残高・出品/購入した商品の一覧(進行中の取引への入口)・換金申請 | 🟢 |
| POST | `/cashout` | 換金申請(申請換金式)。残高から申請額を仮押さえ | 🟢 |
| GET | `/notifications` | 通知一覧(layout.erbのリンク先。既読化もここで) | 🟢 |

### 2.5 運営(admin=trueのユーザーのみ)

| メソッド | パス | 機能 | 差分 |
|---|---|---|---|
| GET | `/admin` | 運営ダッシュボード(審査待ち通報一覧・換金申請一覧・チャージ窓口) | 🟢 |
| POST | `/admin/charge` | 窓口チャージ(対象ユーザー+金額を指定してポイント付与) | 🟢 |
| POST | `/admin/cashout/:id/paid` | 換金申請を支払済みにする(現金手渡し完了の記録) | 🟢 |
| POST | `/admin/judge/:item_id` | 通報の裁定。動画を確認して「返金(status 5→6・購入者へ返金)」または「出品者支持(status 5→4・出品者へ入金)」 | 🟢 |

**ポイントの動きまとめ**(エスクローの実体):
- `buy`: 購入者 −price(この時点で誰の残高でもない=運営預かり状態)
- `complete` / 裁定「出品者支持」: 出品者 +price
- 裁定「返金」: 購入者 +price
- どの経路でも **+と−は必ず一対**。共謀しても合計は増えない(企画書5章#8)

## 5. 設計メモ(実装時の判断材料)

- **番号開示の順序が生命線**: `/transaction/:item_id/receive` は「録画が始まった後」にしか番号を返してはいけない。画面遷移の順序として「受け取り開始ボタン → カメラ起動・録画開始 → 番号表示」を厳守する(企画書4.1の根拠そのもの)
- **アクセス制御は3段階**: ①未ログインは閲覧のみ ②取引ページは当事者(user_id / buyer_id)のみ ③/admin配下はadminのみ。各ルートの冒頭でチェック
- **locker_passは詳細ページに絶対に出さない**: 表示するのはstatus 3以降の購入者本人にのみ、取引ページ内でのみ
- **自分の商品は買えない**: `/item/:id/buy` で user_id == 購入者 を弾く
- **残高不足の購入を弾く**: point < price ならエラー表示
- **換金申請時は残高から即時減算**(仮押さえ)し、却下時に戻す。二重申請での引き出し超過を防ぐ
- **動画ファイル名は推測不能に**(ランダム文字列)。public/videos直下でも実質的に第三者は開けない
- **status遷移は必ず現在値を確認してから更新**(例: depositはstatus==1のときだけ)。URLを直接叩かれても遷移が飛ばないように

## 6. 実装フェーズ分割

### 分割の考え方

- **第1段階「フリマコア」**: カメラ・録画・運営裁定を全部抜いても、「出品→購入→投函→番号で開錠→受取完了」のループはフリマとして完結する。まずこれを動かす
- **第2段階「ScholarShopとしての完成」**: 第1段階のループに「録画ゲート」と「運営」を差し込む。企画書のコアアイデア(録画連動型解錠・裁定・ポイント窓口)はすべてここ

第1段階を作り込んでも第2段階でほぼ壊さないように、**作り直しが発生するのは `/transaction/:item_id/receive` の1ルートだけ**になるよう設計してある。

### 第1段階: フリマコア(15ルート)

| 範囲 | 内容 |
|---|---|
| 認証 | GET/POST `/register`、GET/POST `/login`、GET `/logout` |
| 商品 | GET `/`、GET/POST `/new_item`(画像アップロード込み)、GET `/item/:id`、POST `/item/:id/buy` |
| 取引 | GET `/transaction/:item_id`、POST `chat` / `deposit` / `receive` / `complete` |
| マイページ | GET `/mypage`(残高・自分の取引一覧。取引ページへの動線として必須) |
| DB | マイグレーション1〜3(users / items / chats の変更のみ。新テーブルなし) |
| ビュー | layout / index / item / new_item / login / register / transaction / mypage |
| status | 0→1→2→3→4 のハッピーパスのみ(5・6は使わない) |

**第1段階の仮実装(第2段階で差し替える箇所)**:

| 仮実装 | 第2段階での姿 |
|---|---|
| `receive`で**即座に番号を表示**(録画なし) | 録画開始を確認してから番号表示(録画ゲート挿入)+ `video`アップロード |
| **登録時に初期ポイントを付与**(例: 1000pt。テスト用にこれで経済を回す) | 付与を廃止し、運営窓口チャージ(`/admin/charge`)に置き換え |
| layout.erbの通知リンクは**コメントアウト** | `/notifications` 実装と同時に復活 |
| 通報ボタンなし(取引ページはcompleteのみ) | `report` ボタンと審査フロー追加 |

※ポイントの減算(buy)・入金(complete)・エスクローの考え方は第1段階から本実装。ここは差し替え不要。
